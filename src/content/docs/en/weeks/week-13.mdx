## What is Ralphthon?

**Ralphthon** is the course capstone. Unlike a normal hackathon, the deliverable is not just an app. The deliverable is a **repeatable agentic system** that takes a task, acts, checks itself, logs what happened, and can be replayed.

<Aside type="note" title="Capstone goal">
The final question is not "did AI write a lot of code?" The question is whether humans defined the requirements and boundaries while agents executed under a harness that enforced quality and safety.
</Aside>

### Conditions for a good topic

| Condition | Meaning | Bad example |
|----------|---------|-------------|
| Repeatable | the task appears more than once | one-off demo |
| Verifiable | tests, rubrics, judge, or review can evaluate it | "write something good" |
| Bounded | files, tools, and permissions can be scoped | unrestricted internet automation |
| Recoverable | bad outputs can be rolled back or retried | direct DB mutation |
| Team-splittable | planner/worker/reviewer/operator work can separate | one person does everything |

## Capstone structure through Agent OS Runtime

The design document must map at least five of these seven layers.

| Layer | What to define |
|------|----------------|
| L1 Tool Boundary | allowed and denied tools |
| L2 Skill/Instruction | role instructions, rubrics, prompts |
| L3 Agent Role | Lead, Planner, Worker, Reviewer, Operator |
| L4 Event Store | `.events.jsonl`, replay snapshot |
| L5 Policy Gate | tests, lint, approvals, budget, secret scan |
| L6 Orchestration | handoff sequence, retry, escalation |
| L7 Artifact | PR, report, dashboard, demo, final presentation |

See [Agent OS 7-Layer Architecture](/en/reference/agent-os-runtime/#7-layer-architecture) for the full layer model.

### MVP scope decision

A strong capstone does not start by promising a complete product. Separate what must remain from what can be cut.

| Scope | Example | Decision rule |
|-------|---------|---------------|
| Must have | task packet, single worker loop, deterministic gate, event log | without this, there is no closed loop |
| Should have | reviewer/judge, replay snapshot, simple dashboard | strengthens final evidence |
| Could have | web UI, multi-model router, agent marketplace | only if time remains |
| Won't have | fully autonomous deployment, external account automation, complex delegation | too risky or too expensive |

The Week 13 design document must include `Won't have`. A plan without scope cuts is a wish list, not an execution plan.

### Task Packet Design

```yaml
task_id: capstone-017
objective: "Add retry handling to the GitHub issue importer"
scope:
  files:
    - src/importer/github.py
    - tests/test_github_importer.py
allowed_tools:
  - read_file
  - edit_file
  - run_tests
acceptance:
  - "pytest tests/test_github_importer.py passes"
  - "No network call in unit tests"
  - "Retry count is configurable"
budget:
  max_turns: 6
  max_tokens: 120000
escalation:
  ask_human_if:
    - "API contract must change"
    - "Secret or credential is required"
```

### Architecture Decision Record

Each team must keep at least three ADRs for important choices.

```markdown
# ADR-001: Use vLLM OpenAI-compatible API

## Context
We need to compare local and commercial models through the same harness.

## Decision
Expose local models through vLLM's OpenAI-compatible API and route calls through one client wrapper.

## Consequences
- Good: model providers can be swapped without changing agent code.
- Bad: model-specific tool-call parsers still require configuration.
```

A small closed loop matters more than a large ambition. If the MVP cannot run in Week 14, Week 15 integration will fail. Start with one demo path and one recovery path.

### Risk Register

Risks must be operational, not abstract. Each one needs a trigger, owner, and response.

| Risk | Trigger | Owner | Response |
|------|---------|-------|----------|
| model repeatedly breaks JSON schema | invalid JSON rate > 20% | Agent Engineer | add structured output or repair step |
| no quality evidence | deterministic gate is empty | QA / Operator | write smoke tests and fixtures |
| scope creep | happy path has not passed by Week 14 | Lead | remove all Could have items |
| cost spike | token budget exceeded per run | Harness Engineer | enforce max turns, trim context, cache prefix |
| unstable demo | 1+ failure in 3 repeated runs | QA / Operator | shorten live demo and prepare recording |

This table flows directly into the Week 14 progress report and Week 15 release gate.

## Practicum

<Steps>
1. **Frame the team problem**

   Pick one real recurring task and define the user in one sentence.

2. **Write success criteria**

   Replace "good" with five pass/fail criteria.

3. **Split agent roles**

   Choose only the roles you need: Lead, Planner, Worker, Reviewer, Operator.

4. **Map runtime layers**

   Decide which files, scripts, logs, and policies belong in each layer.

5. **Fix the demo path**

   Choose one happy path and one failure-recovery path for the final demo.
</Steps>
