## Execution Week

The goal this week is a **closed-loop MVP**. One task packet enters the system, an agent acts, a gate decides, and an event log can replay the final state.

<Aside type="tip" title="Ralphthon success rule">
Do not build the large system first. Connect a small happy path end to end, then add failure recovery and observability.
</Aside>

## Day-by-day plan

<Steps>
1. **Day 1: runtime skeleton**

   - Finalize repository structure.
   - Create `tasks/`, `runs/`, `artifacts/`, and `reports/`.
   - Write task packet schema v1.
   - Record the first `run.started` and `run.closed` events.

2. **Day 2: first agent loop**

   - Connect one agent CLI or API client.
   - Let a Worker complete one small task.
   - Run deterministic gates such as tests, lint, or schema checks.
   - Fail clearly without retry.

3. **Day 3: review and retry**

   - Add Reviewer or LLM Judge.
   - Split fail/revise/pass.
   - Apply retry budget and max turns.
   - Verify event log replay.

4. **Day 4: telemetry and dashboard**

   - Connect OpenTelemetry or CSV metrics.
   - Record tokens, latency, pass rate, and judge score.
   - Run the demo path three times.
   - Write the failure runbook.

5. **Day 5: midpoint report**

   - Write the progress report.
   - Cut remaining scope.
   - Fix the final demo path.
   - List the top final-week risks.
</Steps>

## Agent handoff pattern

| Stage | Input | Output |
|------|-------|--------|
| Lead | problem statement | task packet |
| Planner | task packet | implementation plan |
| Worker | plan + repo | patch/artifact |
| Reviewer | patch + criteria | review verdict |
| Gate | verdict + tests | pass/revise/fail |

Use [Agent OS Contracts](/en/reference/agent-os-runtime/#contracts) as the starting point for `lead_directive_v1`, `plan_v1`, `worker_report_v1`, and `review_result_v1`.
</Aside>

## Runbook

| Problem | Immediate action | Long-term fix |
|---------|------------------|---------------|
| Agent edits out-of-scope files | stop run and discard patch | strengthen task packet scope and tool policy |
| Tests keep failing | stop retrying and record reason | split the task smaller |
| Judge emits invalid JSON | schema validation fail | add structured output or repair step |
| Token cost spikes | enforce max turns/tokens | shorten prompt prefix and improve cache hits |
| Merge conflicts | declare file ownership | use branches or worktrees |
| Demo is unstable | prepare recording | shorten the happy path |

## Pivot criteria

Cut scope if any of these are true:

- no end-to-end happy path has passed;
- event logs cannot reconstruct failures;
- no deterministic, judge, or policy gate is automated;
- model connectivity consumes most of the time;
- team members are implementing different goals.

## Mentoring check-in questions

1. What is the demo path?
2. What is the last successful run id?
3. What failed in the last failed run?
4. What remaining scope can be cut?
5. Which metric will appear in the final presentation?

### Week 14 Definition of Done

| Item | Minimum passing standard |
|------|--------------------------|
| Closed loop | one task packet runs end to end |
| Event log | `run.started`, key tool events, and `run.closed` are recorded |
| Gate | at least one deterministic gate runs automatically |
| Replay | final status can be recomputed from the event log |
| Report | team can explain the last successful and failed runs |

If this is not done, Week 15 should focus on completing the loop rather than adding features.
