## Goal of the final presentation

The final presentation is not just a demo. It proves that your team can treat AI systems as engineering systems.

1. What recurring task did you address?
2. How far did the agent execute autonomously?
3. How did the harness control failure?
4. What evidence supports result quality?
5. What would you cut or strengthen next?

<Aside type="note" title="What strong evidence sounds like">
Say: "In this run id, this task packet entered, these gates passed, and this event was recorded." Avoid stopping at "the AI answered this."
</Aside>

### Presentation evidence checklist

Every final presentation claim should have evidence next to it.

| Claim | Required evidence |
|-------|-------------------|
| the system works | live demo or recording + run id |
| it is repeatable | three runs of the same task packet |
| it fails safely | failure-path event log |
| quality was evaluated | deterministic test + judge + human review table |
| cost is understood | token/GPU/operator cost formula |
| next steps are grounded | backlog tied to failure reasons |

Cut claims that do not have evidence.

## Peer review

Peer review is engineering review, not popularity voting. For each team, write:

1. the most convincing design decision;
2. the riskiest failure mode;
3. one gate needed before real use;
4. the clearest metric or evidence in the presentation;
5. one sentence of improvement advice.

## Individual reflection questions

<Steps>
1. **What scope did you cut most meaningfully?**

   Explain why cutting it was an engineering decision, not a failure.

2. **Where did the AI fail repeatedly?**

   Explain what changed in instructions, gates, or context.

3. **What action would have been risky without a harness?**

   Be specific: file writes, API calls, cost, security, or bad decisions.

4. **What is the first improvement in the next version?**

   Prefer observability, evaluation, and reliability before new features.
</Steps>

## Course wrap-up

The course's central sentence is:

> The stronger the model gets, the more important the harness around it becomes.

If your capstone connected task packets, tool boundaries, event logs, policy gates, judges, and telemetry, then you treated AI systems as engineering systems rather than prompts.
