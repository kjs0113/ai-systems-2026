## Theory

### The 2026 Open-Weight Coding Model Landscape

The important change is no longer that open models are "usable." The change is that open-weight models can now compete inside real agent harnesses, where tool calling, long context, serving compatibility, license, and operating cost matter as much as raw coding benchmarks.

<Aside type="note" title="Open source vs. open weights">
Most current models do not publish the full training stack. They publish weights and usage terms. This course uses "open-weight models" when that is the accurate term.
</Aside>

#### Current model comparison

| Model | Release type | Parameters | Context | Strength | Operational note |
|------|--------------|-----------|---------|----------|------------------|
| **Qwen3-Coder-Next** | Apache 2.0 | 80B MoE, 3B active | 256K | agentic coding, tool calling, local development | newer, smaller, faster coding-agent candidate than 480B-A35B |
| **DeepSeek-V4-Pro / Flash** | model-card public | Pro 1.6T/49B active; Flash 284B/13B active | 1M | long-context reasoning, coding, agentic tasks | Pro targets quality; Flash targets efficiency and throughput |
| **GLM-5.1** | MIT | 754B model-card size | agentic long-context work | agentic engineering, coding, terminal tasks | current GLM comparison baseline; vLLM/SGLang supported |
| **MiniMax-M2.7** | model-card public | 229B | long agent workflows | software engineering, search, office work, agentic teams | newer MiniMax agentic candidate than M2.5 |
| **DeepSeek-Coder-V2-Lite** | DeepSeek | 16B/236B family | 128K | single-GPU lab model | useful for teaching; not the frontier comparison point |

#### How to read a model card

For this course, read operational constraints before benchmark tables.

| Step | Question | Reject signal |
|------|----------|---------------|
| License | Can we use it for class, research, or a PoC? | unclear redistribution or service terms |
| Serving support | Does it officially support vLLM, SGLang, or Transformers? | only community patches exist |
| Tool format | How are tool calls, JSON, and reasoning outputs represented? | every model needs an ad hoc parser |
| Context policy | Is the usable context smaller than the advertised maximum? | no realistic GPU plan for long context |
| Failure behavior | Are OOM, invalid JSON, refusals, and timeouts distinguishable? | failures cannot drive retry decisions |

"Current" in this table means current for the lecture date. The capstone decision must still come from reproduced runs using the same task packets and harness.

#### Qwen3-Coder

Qwen3-Coder-Next is an 80B total / 3B active MoE model with 256K context and explicit agentic coding/tool-use positioning. The operational point is not just coding quality. It is a practical candidate for local or institutional agentic coding harnesses because it combines long repository context, tool-use behavior, and vLLM/SGLang deployment support.

#### DeepSeek-V4

DeepSeek-V4-Pro and DeepSeek-V4-Flash are the current DeepSeek V4 comparison points in this course. Pro is the quality and long-context reasoning candidate; Flash is the efficiency and throughput candidate.

| Family | Use in this course |
|--------|--------------------|
| DeepSeek-Coder-V2-Lite | teaching model that can run on smaller lab servers |
| DeepSeek-V4-Pro | current DeepSeek frontier comparison baseline |
| DeepSeek-V4-Flash | cost and throughput comparison baseline |
| earlier DeepSeek models | long-context baseline and historical comparison |

#### GLM-5.1 and MiniMax-M2.7

GLM-5.1 is the current GLM comparison point for agentic engineering, coding, and tool workflows. MiniMax-M2.7 is the current MiniMax candidate for long-horizon agentic workflows across coding, search, and office-like tasks. Treat their model-card benchmark numbers as vendor claims until the same tasks are reproduced inside our harness.

<Aside type="caution" title="How to read benchmarks">
SWE-bench, Terminal-Bench, and BrowseComp scores depend heavily on scaffold, prompt, tool policy, and retry budget. Use public scores to choose candidates, not to declare a winner before lab reproduction.
</Aside>

### Model choice is harness choice

The deployment decision has three realistic options.

<CardGrid>
  <Card title="Commercial API" icon="external">
    Use Claude, GPT, or Gemini APIs directly. Quality and convenience are high, while data boundaries, cost forecasting, and rate limits need management.
  </Card>
  <Card title="Cloud Agent" icon="rocket">
    Use Codex Web/App, Claude Code on the Web, or GitHub Agent HQ. Strong for asynchronous coding, review, and GitHub workflows.
  </Card>
  <Card title="Local Serving" icon="laptop">
    Serve an OpenAI-compatible endpoint with vLLM or SGLang. Data control and predictable GPU cost improve, but operational complexity increases.
  </Card>
</CardGrid>

| Criterion | Commercial API | Cloud agent | Local serving |
|-----------|----------------|-------------|---------------|
| Startup speed | Very fast | Fast | Slow |
| Data control | Low-medium | Medium | High |
| Long-term cost | Usage-based | Seat/usage-based | GPU fixed cost |
| Latest model access | Fast | Fast | After weight release |
| Ops complexity | Low | Medium | High |
| Course value | API design | Workflow design | MLOps and infra |

### Why OpenAI-compatible APIs matter

vLLM and SGLang usually expose OpenAI-compatible chat endpoints. That lets the harness swap model providers without rewriting the agent runtime.

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="local-dev-token",
)

response = client.chat.completions.create(
    model="Qwen/Qwen3-Coder-Next",
    messages=[
        {"role": "system", "content": "You are a careful coding agent."},
        {"role": "user", "content": "Add tests for the parser edge cases."},
    ],
    temperature=0.2,
)
```

| Layer | Responsibility |
|------|----------------|
| Agent CLI | File edits, commands, approvals, workflow |
| Gateway | Model routing, budget, rate limits, audit |
| Serving | Batching, KV cache, speculative decoding, GPU scheduling |
| Model | Token generation, tool-call JSON, reasoning output |

#### Model adapter checklist

OpenAI compatibility is the common starting point, not the whole abstraction. A useful model adapter must absorb:

| Item | Implementation rule |
|------|---------------------|
| Model alias | stable names such as `local-coder`, `qwen3-coder`, and `glm-5.1` |
| Message template | system/developer/user/tool messages aligned with each model's chat template |
| Tool parser | function-call JSON, XML-like blocks, and plain-text plans parsed per model |
| Retry policy | timeout, invalid JSON, context overflow, and refusal recorded separately |
| Token accounting | prompt, completion, and cache tokens linked to run id |
| Evaluation hook | same task packet can run against commercial and local models |

Without this checklist, "model swapping" is a comment in the code rather than an operational property.

## Practicum

### Deploy an open-weight model with vLLM

<Steps>
1. **Check the environment**

   ```bash
   nvidia-smi
   uv venv .venv
   source .venv/bin/activate
   uv pip install vllm openai
   ```

2. **Choose a lab model**

   <Tabs>
     <TabItem label="Teaching default">
       ```bash
       vllm serve deepseek-ai/DeepSeek-Coder-V2-Lite-Instruct \
         --served-model-name local-coder \
         --max-model-len 32768 \
         --port 8000
       ```
     </TabItem>
     <TabItem label="Qwen3-Coder-Next">
       ```bash
       vllm serve Qwen/Qwen3-Coder-Next \
         --served-model-name qwen3-coder \
         --tensor-parallel-size 2 \
         --max-model-len 65536 \
         --port 8000
       ```
     </TabItem>
     <TabItem label="GLM-5.1">
       ```bash
       vllm serve zai-org/GLM-5.1 \
         --served-model-name glm-5.1 \
         --tensor-parallel-size 4 \
         --enable-auto-tool-choice \
         --port 8000
       ```
     </TabItem>
   </Tabs>

3. **Test the OpenAI-compatible endpoint**

   ```bash
   curl http://localhost:8000/v1/chat/completions \
     -H "Content-Type: application/json" \
     -d '{
       "model": "local-coder",
       "messages": [
         {"role": "user", "content": "Implement a stable topological sort in Python."}
       ],
       "temperature": 0.2
     }'
   ```

4. **Compare identical tasks**

   Run the same five tasks against one commercial model and one local model: bug fix, refactor, test generation, docs update, and CLI feature.

5. **Measure cost and throughput**

   ```bash
   python -m vllm.benchmarks.benchmark_serving \
     --backend openai-chat \
     --base-url http://localhost:8000 \
     --model local-coder \
     --num-prompts 100 \
     --request-rate 4
   ```
</Steps>
