## Theory

### The inference server is the operating system for the model

Deploying an LLM is not just loading weights. Service quality depends on the serving layer.

| Operational problem | Serving responsibility |
|--------------------|------------------------|
| GPU memory pressure | KV cache management, quantization, tensor parallelism |
| Request latency | continuous batching, chunked prefill, speculative decoding |
| Repeated prompt cost | automatic prefix caching |
| Long-context bursts | prefill/decode separation and context policy |
| Failure analysis | metrics, traces, request logs, token accounting |

vLLM wraps these concerns behind an OpenAI-compatible API so that the agent harness can keep one interface while changing models.

### PagedAttention: the starting point

Traditional inference reserves large contiguous KV cache regions. Different request lengths cause fragmentation. vLLM's **PagedAttention** breaks KV cache into blocks, similar to virtual memory pages, and links only the blocks a request needs.

PagedAttention is the baseline. The 2026 operations question is **which optimization to enable for which workload**.

### Five optimization levers

<CardGrid>
  <Card title="Automatic Prefix Caching" icon="seti:config">
    Reuses KV cache for repeated system prompts, tool schemas, and project instructions. Ralph loops benefit when the static prefix is long.
  </Card>
  <Card title="Chunked Prefill" icon="seti:clock">
    Splits long prefill work into chunks so a single long request does not block the whole queue.
  </Card>
  <Card title="Speculative Decoding" icon="rocket">
    Uses a draft model or n-gram speculation to propose tokens that the target model verifies, reducing latency.
  </Card>
  <Card title="Disaggregated Prefill" icon="seti:database">
    Separates prefill and decode onto different workers or GPUs for mixed long-context workloads.
  </Card>
  <Card title="Structured Outputs" icon="seti:json">
    Enforces JSON schemas, tool calls, or reasoning outputs at the serving boundary to reduce parser failures.
  </Card>
</CardGrid>

<Aside type="tip" title="Ralph loops and prefix caching">
Ralph loops repeatedly send the same PROMPT.md, project instructions, and tool schemas. The longer that stable prefix is, the more valuable automatic prefix caching becomes.
</Aside>

### Choose optimizations by workload

Turning on every option is not optimization. Identify the bottleneck first, then enable one lever at a time.

| Workload | Main bottleneck | First option to try | Metric to watch |
|----------|-----------------|---------------------|-----------------|
| long repository context + short patch | prefill cost | prefix caching, chunked prefill | TTFT, cache hit ratio |
| many short requests | batching efficiency | continuous batching, tuned `max_num_seqs` | throughput, queue time |
| mixed long summaries and short coding requests | long prefill blocking | chunked prefill, disaggregated prefill | p95 latency, queue time |
| interactive coding assistant | first-token delay | speculative decoding, smaller max tokens | TTFT, perceived latency |
| strict JSON or tool output | parser failure | structured outputs and tool parser config | invalid JSON rate |

The lab report should not say only "we enabled option X." It should show how the bottleneck metric changed before and after.

### Measure throughput and latency together

| Metric | Meaning | Bad signal |
|--------|---------|-----------|
| TTFT | Time to first token | long prefill blocks the queue |
| TPOT | Time per output token | low decode utilization |
| Throughput | tokens/sec | weak batching or memory fragmentation |
| Queue time | waiting before execution | excessive concurrency |
| Cache hit ratio | prefix reuse | unstable prompt assembly |
| Error rate | timeout/OOM/parser failures | capacity or format mismatch |

### Admission control and context budget

When a serving system slows down, accepting every request can make the whole classroom unusable. Production serving decides before execution whether a request should run.

| Control | Example policy |
|---------|----------------|
| Maximum input length | course-level limit below `max_model_len` |
| Concurrency | separate team queues from the global queue |
| Priority | demo and replay requests before experimental batches |
| Timeout | record prefill timeout and decode timeout separately |
| Fallback | reroute to a smaller local model or commercial API |

From the Agent OS Runtime perspective, admission control is a policy gate. Rejecting a request can be a successful boundary-preserving event, not merely a failure.

### Multi-tenant serving

DGX classroom operations are more stable when teams have clear resource boundaries.

```bash
# Team A: teaching model
CUDA_VISIBLE_DEVICES=MIG-GPU-a vllm serve deepseek-ai/DeepSeek-Coder-V2-Lite-Instruct \
  --served-model-name team-a-coder \
  --max-model-len 16384 \
  --port 8001

# Team B: larger model
CUDA_VISIBLE_DEVICES=MIG-GPU-b vllm serve Qwen/Qwen3-Coder-30B-A3B-Instruct \
  --served-model-name team-b-coder \
  --max-model-len 32768 \
  --port 8002
```

<Aside type="caution" title="MIG and tensor parallelism">
MIG is useful for hardware isolation. It is not the default answer for splitting one large model across slices. For large tensor-parallel models, prefer full GPUs or a planned multi-GPU setup.
</Aside>

### Observability first

Every serving request should produce at least:

```json
{
  "request_id": "run-20260512-001",
  "model": "local-coder",
  "prompt_tokens": 1842,
  "completion_tokens": 419,
  "ttft_ms": 820,
  "tpot_ms": 34,
  "cache_hit": true,
  "finish_reason": "stop"
}
```

The serving layer records tokens and latency. The agent runtime records tool calls, approvals, tests, and replay state.

## Practicum

<Steps>
1. **Run a baseline server**

   ```bash
   vllm serve deepseek-ai/DeepSeek-Coder-V2-Lite-Instruct \
     --served-model-name local-coder \
     --max-model-len 32768 \
     --port 8000
   ```

2. **Compare prefix caching on/off**

   Send 20 requests with the same system prompt and repository summary. Compare TTFT, throughput, and cache hits.

3. **Compare chunked prefill**

   Mix long repository summaries with short code requests and measure queue time.

4. **Write an OpenAI-compatible client**

   ```python
   from openai import OpenAI
   import time

   client = OpenAI(base_url="http://localhost:8000/v1", api_key="local")
   started = time.perf_counter()
   response = client.chat.completions.create(
       model="local-coder",
       messages=[{"role": "user", "content": "Write a pytest fixture example."}],
       temperature=0.2,
   )
   print(time.perf_counter() - started, response.choices[0].message.content[:200])
   ```

5. **Prepare the dashboard**

   Record `ttft_ms`, `tokens/sec`, `error_rate`, and `cache_hit`. Week 12 connects these values to traces and event logs.
</Steps>
