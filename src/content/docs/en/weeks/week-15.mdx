## Goal of Integration Week

Week 15 is not a feature week. It is the week to turn the closed loop into a **reliable demo and evidence-backed report**.

<Aside type="caution" title="No new features by default">
Adding a new agent role, model, or UI in Week 15 usually reduces final presentation reliability. Add only what stabilizes the demo path.
</Aside>

### Feature freeze rule

At the start of Week 15, each team writes `freeze.md`.

| Item | Content |
|------|---------|
| Demo path | one happy path that must appear in the presentation |
| Recovery path | one safe failure-handling path |
| Frozen features | features that will no longer change |
| Allowed fixes | bug fixes, test stabilization, documentation, metrics |
| Explicit cuts | features excluded from the final presentation |

After freeze, a change is allowed only when it improves release gate reliability.

## Release Gate

| Gate | Criteria | Evidence |
|------|----------|----------|
| Build/Test | `make test` or equivalent passes | CI log or terminal capture |
| End-to-end | same task packet passes three times | three run ids |
| Replay | final state reconstructed from event log | `replay_snapshot.json` |
| Evaluation | deterministic and judge results recorded | evaluation table |
| Security | no secrets or out-of-scope writes | checklist |
| Demo | backup materials work without network | recording/screenshots |

### Release readiness review

Before final rehearsal, run a 20-minute internal release review.

1. Run `make test` or the equivalent command and capture the result.
2. Run the same task packet three times and record the run ids.
3. Pick the most recent failed run and explain the failure reason.
4. Confirm that `replay_snapshot.json` matches the event log.
5. Put a source run id next to every metric in the slides.
6. Cross-check secrets, credentials, and out-of-scope writes.

Issues found here outrank new feature work.

## Integration test matrix

<Tabs>
  <TabItem label="Happy Path">
    ```text
    task packet -> planner -> worker -> tests pass -> judge pass -> artifact saved
    ```
    Run at least three times.
  </TabItem>
  <TabItem label="Failure Path">
    ```text
    invalid task -> schema validation fail -> no file write -> run.closed(failed)
    ```
    Show that the system stops safely on bad input.
  </TabItem>
  <TabItem label="Recovery Path">
    ```text
    tests fail -> reviewer says revise -> retry once -> pass or closed(failed)
    ```
    Show that retry budget prevents infinite loops.
  </TabItem>
</Tabs>

```text
commercial_api_cost = input_tokens * input_price + output_tokens * output_price
local_gpu_cost = gpu_hour_price * elapsed_seconds / 3600
operator_cost = human_minutes * hourly_rate / 60
```

<Aside type="tip" title="Cost trap">
Local models may look cheap per token, but GPU idle time, model loading, operator time, and retries can dominate end-to-end task cost.
</Aside>

|------|---------|----------|
| 2 min | Problem | user, recurring task, risk boundary |

| 2 min | Reflection | failures, scope cuts, next steps |

## Rehearsal checklist

<Steps>
1. **Cold start rehearsal**

   Start from nothing and prepare the presentation environment.

2. **Time-box rehearsal**

   Present within 15 minutes and finish the demo by minute 12.

3. **Failure injection**

   Run one deliberately bad task and confirm safe failure.

4. **Evidence check**

   Link every metric to a run id, event log, or dashboard screenshot.

5. **Speaker handoff**

   Define transitions between speakers so the presentation can continue if one person is absent.
</Steps>
