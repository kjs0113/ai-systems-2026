# MIG 프로파일 분할 분석 보고서

## 1. H100 80GB MIG 프로파일 개요

NVIDIA H100 80GB GPU는 총 132개의 SM(Streaming Multiprocessor)과 80GB HBM3 메모리를 탑재하며,
MIG(Multi-Instance GPU) 기술을 통해 하드웨어 수준에서 독립된 GPU 인스턴스로 분할할 수 있다.

## 2. 구성 비교

### 구성 A: 1g.10gb × 7

| 항목 | 값 |
|------|-----|
| 프로파일 | 1g.10gb |
| 인스턴스 수 | 7 |
| SM 수 (인스턴스당) | 16 SM (≈ 전체의 1/8) |
| 메모리 (인스턴스당) | 10 GB HBM3 |
| 총 사용 메모리 | 70 GB (10 GB × 7, 나머지 약 10GB는 시스템 예약) |
| L2 캐시 (인스턴스당) | 1/8 슬라이스 (약 6.25 MB) |
| 최대 인스턴스 | 7 (H100 최대) |

### 구성 B: 3g.40gb × 2 + 1g.10gb × 1

| 항목 | 3g.40gb | 1g.10gb |
|------|---------|---------|
| 인스턴스 수 | 2 | 1 |
| SM 수 (인스턴스당) | 50 SM (≈ 전체의 3/8) | 16 SM |
| 메모리 (인스턴스당) | 40 GB | 10 GB |
| L2 캐시 (인스턴스당) | 3/8 슬라이스 (약 18.75 MB) | 1/8 슬라이스 |
| 총 SM | 100 + 16 = 116 SM | |
| 총 메모리 | 80 + 10 = 90 GB (실제 약 80GB 한도 내 배치) | |
| 최대 인스턴스 | 3 (2 + 1) | |

## 3. 비교 분석

### 3.1 SM(연산 능력) 관점

| 기준 | 구성 A (1g.10gb × 7) | 구성 B (3g.40gb × 2 + 1g.10gb × 1) |
|------|----------------------|-------------------------------------|
| 인스턴스당 SM | 16 SM | 50 SM / 16 SM |
| 총 활성 SM | 112 SM (16 × 7) | 116 SM (50 × 2 + 16) |
| 단일 워크로드 최대 SM | 16 SM | 50 SM |

- **구성 A**는 인스턴스당 SM이 적어 대규모 행렬 연산이나 LLM 추론에 병목 발생
- **구성 B**의 3g.40gb 인스턴스는 50 SM으로 중형 모델 추론/학습 가능

### 3.2 메모리 관점

| 기준 | 구성 A | 구성 B |
|------|--------|--------|
| 인스턴스당 최대 메모리 | 10 GB | 40 GB / 10 GB |
| LLM 로드 가능 모델 크기 | ~7B 4-bit (≈5GB) | ~30B 4-bit (≈17GB) / ~7B 4-bit |
| 메모리 대역폭 분할 | 1/7 슬라이스 | 3/7 슬라이스 / 1/7 슬라이스 |

- **구성 A**: 10GB 한도로 Llama-3-8B 4-bit (약 4.5GB) 수준까지 가능. 13B 이상은 불가
- **구성 B**: 3g.40gb에서 Llama-3-70B 4-bit (약 35GB)도 적재 가능

### 3.3 최대 인스턴스(동시 사용자) 관점

| 기준 | 구성 A | 구성 B |
|------|--------|--------|
| 총 인스턴스 수 | 7 | 3 |
| 동시 사용자 | 최대 7명 | 최대 3명 |
| 격리 수준 | 하드웨어 격리 × 7 | 하드웨어 격리 × 3 |

- **구성 A**: 학생 실습 환경처럼 다수의 사용자에게 독립 환경을 제공할 때 최적
- **구성 B**: 사용자 수는 적지만 고성능 워크로드가 필요한 경우 적합

## 4. 장단점 요약

### 구성 A: 1g.10gb × 7

**장점:**
- 최대 7명 동시 사용 → 학생 실습 환경에 이상적
- 하드웨어 격리로 noisy neighbor 문제 없음
- 각 인스턴스가 예측 가능한 성능(QoS) 보장
- 리소스 활용률 극대화 (유휴 GPU 슬라이스 최소화)

**단점:**
- 인스턴스당 10GB → 중형 이상 모델 로드 불가
- 16 SM → 대규모 병렬 연산(행렬 곱 등) 성능 제한
- L2 캐시 1/8 → 캐시 히트율 저하로 메모리 바운드 워크로드 성능 감소

### 구성 B: 3g.40gb × 2 + 1g.10gb × 1

**장점:**
- 3g.40gb 인스턴스에서 중대형 모델(30B 4-bit) 추론 가능
- 50 SM으로 fine-tuning, 배치 추론 등 무거운 작업 처리 가능
- 유연한 혼합 구성: 대형 작업 2개 + 경량 작업 1개 동시 실행
- L2 캐시 3/8로 캐시 효율성 향상

**단점:**
- 총 3명만 동시 사용 가능 → 다수 학생 실습에 부적합
- GPU 리소스의 비대칭 분배 → 1g.10gb 사용자와의 형평성 문제
- 구성 변경 시 모든 MIG 인스턴스를 재생성해야 함 (다운타임 발생)

## 5. 사용 시나리오 권장

| 시나리오 | 권장 구성 | 이유 |
|----------|-----------|------|
| 대학 수업 (7명 동시 실습) | A (1g.10gb × 7) | 동시 사용자 수 극대화, 공정한 리소스 분배 |
| 연구실 (소규모, 대형 모델) | B (3g.40gb × 2 + 1g.10gb × 1) | 중대형 모델 로드 가능 |
| 추론 서비스 (혼합 워크로드) | B | 대형/소형 요청을 인스턴스별로 라우팅 |
| CI/CD 테스트 파이프라인 | A | 병렬 테스트 실행 수 극대화 |

## 6. 결론

교육 환경에서는 **구성 A (1g.10gb × 7)**가 적합하다. 7명의 학생이 동시에 독립된 GPU 인스턴스에서
실습할 수 있으며, Llama-3-8B 4-bit 양자화 모델까지 로드 가능하다.
연구/프로덕션 환경에서 대형 모델이 필요하면 **구성 B**가 합리적이다.
MIG의 핵심 가치는 하드웨어 수준 격리를 통한 QoS 보장이며, 프로파일 선택은 워크로드 특성과
동시 사용자 수의 트레이드오프를 고려하여 결정해야 한다.
