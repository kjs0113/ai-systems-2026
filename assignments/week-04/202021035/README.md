# Lab 04 설명서 — Ralph 루프 하네스 설계

이 저장소에는 **Lab 04: Ralph 루프 구현**을 위한 샘플 하네스가 포함되어 있습니다. `harness.sh` 스크립트는 단순화된 Ralph 패턴을 구현하여 반복적으로 에이전트를 실행하고, 실패에서 학습하며, 성공할 때까지 재시도합니다. 아래에서 주요 설계 결정과 테스트 시간 컴퓨트 스케일링(Test‑Time Compute Scaling)과의 연결을 설명합니다.

## 하네스 구성요소

| 파일 | 설명 |
|-----|-----|
| `tasks.json` | 수행해야 할 태스크 목록을 정의합니다. 각 항목은 식별자, 제목, `passes` 플래그를 갖습니다. |
| `harness.sh` | 메인 루프를 실행하는 Bash 스크립트입니다. 현재 상태를 읽고, 아직 완료되지 않은 태스크를 순서대로 실행하며, 결과를 기록합니다. 백프레셔(backpressure)와 가비지 컬렉션(garbage collection)을 구현합니다. |
| `run_task.py` | 실제 에이전트를 대신해 태스크를 실행하는 도우미 스크립트입니다. 초기 반복에서 일부 태스크를 실패하도록 설정하여 Ralph 패턴의 “실패→학습→재시도”를 시뮬레이션합니다. |
| `PROMPT.md` | 세 개의 프로그래밍 태스크에 대한 상세 명세와 수락 조건을 담고 있습니다. |
| `AGENTS.md` | 각 반복에서 얻은 학습을 누적해 기록하는 문서입니다. 에이전트는 이 파일을 읽어 다음 반복에서 동일한 실수를 하지 않도록 합니다. |
| `loop_log.txt` | 루프 실행 중의 터미널 로그를 저장합니다. 실패→학습→재시도 과정이 명확히 드러나도록 설계되어 있습니다. |
| `metrics.csv` | 반복별로 사용된 토큰 수와 실행 시간을 기록하여 컴퓨트 소비량을 분석할 수 있게 합니다. |

## 설계 결정

### 상태 분리와 신선한 컨텍스트

Ralph 루프의 핵심 원칙은 **매 반복마다 새로운 컨텍스트**를 사용하는 것입니다. 이 구현에서는 작업 상태를 `state/progress.json` 파일에 보존하고, 각 반복마다 이를 읽어 미완료 태스크를 결정합니다. 진행 상황만을 디스크에 기록하고 에이전트 컨텍스트 자체는 매번 재생성함으로써, 오래된 컨텍스트가 이후 반복에 영향을 미치지 않도록 합니다. 이는 Ralph 원칙의 “Fresh Context Is Reliability”와 일치합니다【350383466409196†L459-L486】.

### 백프레셔 구현

무한 반복과 리소스 과다 사용을 방지하기 위해 **백프레셔(backpressure)** 메커니즘을 도입했습니다. `harness.sh`는 연속된 실패 횟수를 추적하고, 실패가 세 번 이상 누적되면 다음 반복 전 1초간 일시 정지합니다. 실제 시스템에서는 대기 시간, 큐 길이, 메모리 사용량 등을 고려해 동적으로 조정할 수 있지만, 본 하네스는 개념을 단순하게 보여주기 위해 시간 지연으로 구현했습니다. 이를 통해 에이전트가 지속적으로 잘못된 출력만 생성하는 상황을 제어할 수 있습니다【350383466409196†L475-L486】.

### 가비지 컬렉션

에이전트가 여러 번 반복되면 상태 파일과 로그가 빠르게 누적될 수 있습니다. `harness.sh`는 각 반복 후 `archive/` 디렉터리에 진행 상태 스냅샷을 저장하고, 보존할 스냅샷의 개수를 `MAX_ARCHIVES`로 제한합니다. 이 과정을 통해 오래된 상태를 자동으로 정리하여 디스크 공간을 관리합니다. 이는 Ralph 루프에서 언급되는 ‘엔트로피 관리’와 ‘디스크는 상태, Git은 메모리’라는 개념과 관련이 있습니다【350383466409196†L475-L486】.

### 학습 기록과 피드백 루프

각 반복에서 성공과 실패로부터 얻은 교훈은 `AGENTS.md`에 한 줄씩 추가됩니다. 다음 반복에서 에이전트는 이 문서를 참고해 동일한 실수를 피하거나 발견된 패턴을 재사용합니다. 이러한 **피드백 루프**가 없다면 루프는 발전하지 못하고 무한히 실패를 반복하게 됩니다. Ralph 프로젝트 또한 `progress.txt`와 `AGENTS.md`를 사용해 같은 목적을 달성합니다【806923215441700†L462-L470】.

### 컴퓨트 메트릭 수집과 테스트 시간 스케일링

`metrics.csv`에는 각 태스크 실행에 소요된 시간(밀리초)과 사용된 토큰 수(임의의 시뮬레이션 값)가 저장됩니다. 실제 에이전트에서는 언어 모델 호출에 따라 토큰 비용과 시간이 크게 달라질 수 있으며, 테스트 시간 컴퓨트 스케일링은 이러한 자원 소비를 관리하는 중요한 기법입니다. 메트릭을 수집함으로써:

1. **예산 관리**: 반복당 평균 토큰 수를 계산해 전체 작업에 필요한 토큰 예산을 추정할 수 있습니다.
2. **성능 최적화**: 특정 태스크가 지나치게 많은 토큰을 사용하거나 시간이 오래 걸리는 경우, 프롬프트를 조정하거나 태스크를 세분화하는 등의 최적화를 할 수 있습니다.
3. **스케일링 전략 수립**: 작은 예제에서 수집한 메트릭을 기반으로 더 큰 프로젝트에 확장할 때 필요한 리소스를 예측할 수 있습니다.

따라서 본 하네스는 단순한 과제 시나리오에서도 실험적 토큰/시간 데이터를 수집하도록 설계되었습니다. 이러한 데이터는 테스트 시간 컴퓨트 스케일링 전략을 세울 때 핵심적인 근거 자료로 활용됩니다.

## 실행 방법

1. `harness.sh`를 실행하기 전에 작업 디렉터리에서 실행 권한을 부여합니다. 예:

   ```bash
   chmod +x harness.sh
   ```

2. 루프를 실행합니다. 인자로 최대 반복 횟수를 지정할 수 있으며, 기본값은 10입니다.

   ```bash
   ./harness.sh 5
   ```

3. 실행 결과는 `loop_log.txt`에 기록되고, 반복별 메트릭은 `metrics.csv`에서 확인할 수 있습니다. 모든 태스크가 `passes: true`가 될 때 루프는 종료됩니다.

## 향후 개선 방향

- **실제 에이전트 통합**: 현재 `run_task.py`는 실패를 시뮬레이션합니다. 이를 코딩 에이전트(예: Claude Code, Amp)와 통합하면 실제 프로젝트를 자동화할 수 있습니다.
- **동적 태스크 분할**: 실패가 일정 횟수 이상 지속될 경우 태스크를 더 작은 단위로 분할하여 재시도하는 로직을 추가하면 생산성을 높일 수 있습니다.
- **고급 백프레셔**: 리소스 사용량이나 실패 패턴에 따라 지능적으로 루프 속도를 조절하는 알고리즘을 도입할 수 있습니다.
- **추가 메트릭**: 메모리 사용량, 프롬프트 길이 등 다른 지표를 수집하여 더 정밀한 스케일링 분석을 수행할 수 있습니다.
